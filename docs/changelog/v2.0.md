# Dataphyre 2.0: v1 to v2 Migration Notes

Dataphyre 2.0 is the public-release reshaping of the v1 runtime. The change is
large enough that it should be treated as a major upgrade, not a patch-level
drop-in replacement.

This document summarizes the upgrade from:

- v1 baseline:
  [`a682534470207a31460a5c6626b760d792647e3b`](https://github.com/jeremie5/dataphyre/commit/a682534470207a31460a5c6626b760d792647e3b)
  (`Let there be light, again`, 2026-04-26)
- v2 release:
  [`6adbcbc56000c24be4c94199a9beaa2a4d24ecb3`](https://github.com/jeremie5/dataphyre/commit/6adbcbc56000c24be4c94199a9beaa2a4d24ecb3)
  (`2.0 release`, 2026-06-25)

Measured from v1 to v2 after the public history cleanup, the repository changed
`923` files with roughly `329300` insertions and `10356` deletions. The full
compare is available at
[`a682534...6adbcbc`](https://github.com/jeremie5/dataphyre/compare/a682534470207a31460a5c6626b760d792647e3b...6adbcbc56000c24be4c94199a9beaa2a4d24ecb3).

## What Changed

### Public release boundary

Dataphyre 2.0 formalizes the repository as an embedded runtime distribution:

- `runtime/` remains the reusable engine boundary.
- `config/`, `plugins/`, generated state, and bootstrap files are treated as
  install-local shell concerns.
- public documentation moved under `docs/`.
- package artifacts can include `RELEASE_MANIFEST.json`.
- local install files, generated state, and application-owned integrations stay
  outside baseline Dataphyre runtime dependencies.

The 2.0 release is now explicit about what is framework source and what is
application-owned behavior.

### Documentation and package metadata

The v1 root-level docs were reorganized into a public docs tree. Dataphyre 2.0
adds or expands:

- `docs/README.md`
- `docs/MODULES.md`
- `docs/AGENTIC_ENTERPRISE.md`
- `docs/AGENTS.md`
- `docs/RELEASE_MANIFEST.md`
- `docs/RELEASE_MANIFEST.schema.json`
- `docs/CONFIGURATION.md`
- `docs/PACKAGE.md`
- `docs/STABILITY.md`
- `docs/SECURITY.md`
- `docs/SUPPORT.md`
- `docs/THIRD_PARTY_NOTICES.md`

Composer metadata now documents Dataphyre as `dataphyre/dataphyre`, requires
PHP 8.1 or newer, declares MIT licensing, and points consumers to the explicit
runtime bootstrap instead of implying that `vendor/autoload.php` alone is the
runtime entrypoint.

### Release manifest and verification

Prepared 2.0 exports ship `RELEASE_MANIFEST.json`, a non-sensitive attestation
for release consumers and automation. It includes:

- public module inventory
- bundled component inventory
- packaged file byte counts and SHA-256 hashes
- deterministic export tree hash
- package boundary categories
- verification provenance
- app-agent and maintainer boundary metadata

Package validation was expanded to check Composer metadata, package-boundary
hygiene, Markdown links, JSON fixtures, PHP linting, MIT/SPDX headers, fragile
PHP self-load guards, MCP self-tests, and live MCP stdio validation.

### Agentic and MCP surface

Dataphyre 2.0 introduces a much larger MCP surface for AI-assisted application
work. The default MCP audience is application agents building apps with
Dataphyre, not agents modifying Dataphyre internals.

For ordinary app work, start with:

```text
dataphyre_app_builder_plan_generate
```

Use `payload_profile=compact` first, then open larger detail pages only when the
next body is needed. Maintainer-only release validation, broad verification,
unsafe MCP mode, and hot-path benchmarks are not ordinary app-agent ceremony.

### Framework modules

2.0 greatly expands and documents the public module surface. The most important
additions or expansions include:

- `mcp`: Dataphyre-aware Model Context Protocol server, inspection, planning,
  source tools, app-builder plans, workflow handoffs, and validation helpers.
- `panel`: generated control surfaces, resources, forms, tables, navigation,
  rendering, uploads, themes, package workflows, and regression harnesses.
- `mvc`: controllers, route groups, middleware, sessions, validators, signed
  URLs, service providers, route model binding, and regression tooling.
- `permission`: semantic authorization rules, catalogs, audits, conditions,
  snapshots, Panel integration, and Laravel adapter hooks.
- `reactor`: server-driven component lifecycle, signed snapshots, actions,
  effects, model binding, validation, testing, and client assets.
- `storage`: file storage abstraction for local disks, S3-compatible storage,
  policy wrappers, integrity wrappers, quotas, retention, tagging, encryption,
  and Vestra aliases.
- `mailer`: provider-backed email delivery for SMTP, AWS SES, Brevo,
  Cloudflare, Mailgun, Postmark, Resend, SendGrid, logging, scheduling, and
  delivery results.
- `sql`: stronger query, repository, table, transaction, mutation, and
  scaffolding APIs.
- `templating`: richer template contracts, asset policy, manifests, binding
  contexts, cached bindings, SQL/search bindings, and rendering state.

The module index now labels modules as `core`, `optional`, `adapter`, `legacy`,
or `experimental`.

## Upgrade Guidance

### 1. Treat the upgrade as a replacement

Do not overlay the 2.0 tree on top of a v1 checkout. Start from a clean 2.0
release tree, then reapply local install files and application code deliberately.

Keep these local:

- `flight_sheet.php`
- `index.php`
- `config/*.php`
- `plugins/pre_init/*.php`
- `plugins/post_init/*.php`
- generated `cache/` and `logs/`
- application-owned adapters and product code

Use `flight_sheet.example.php`, `index.example.php`, and `config/*.example.php`
as templates instead of copying old install secrets forward.

### 2. Re-check module assumptions

Review `docs/MODULES.md` before enabling modules. Several modules that were
present in v1 are now application-owned rather than baseline Dataphyre modules.
If an app depended on a removed public module, migrate that behavior into
application code, a plugin, a callback, a dialback, or an app-owned adapter.

### 3. Keep app behavior outside runtime internals

Dataphyre 2.0 makes the extension boundary explicit. Prefer:

- install config
- dialbacks
- callbacks
- plugin hooks
- MCP metadata
- application-owned adapters
- reusable runtime modules only when the behavior belongs in Dataphyre itself

Framework-internal edits should be reserved for reusable Dataphyre work, public
API changes, framework bugs, performance work, or documented runtime capability
changes.

### 4. Update automation and docs links

If scripts or docs referenced root-level v1 documentation files such as
`README.md`, `MODULES.md`, or `CHANGELOG.md`, update them to the 2.0 docs
layout.

Common replacements:

| v1 path | v2 path |
|---|---|
| `README.md` | `docs/README.md` |
| `CHANGELOG.md` | `docs/CHANGELOG.md` |
| `MODULES.md` | `docs/MODULES.md` |

Package artifacts can include `RELEASE_MANIFEST.json` at the package root.

### 5. Run focused verification

For an application using Dataphyre, verify the affected app or module directly.
Framework release validation should stay proportionate to the files and runtime
surface being changed.

## Compatibility Notes

- PHP 8.1 or newer is required.
- The stable entrypoint remains the explicit runtime bootstrap:

```php
<?php

require __DIR__.'/runtime/bootstrap.php';
```

- Composer metadata identifies the package, but the runtime is still best
  treated as an embedded distribution, not a pure Composer autoload library.
- Product-specific adapters are application-owned integrations, not baseline
  dependencies.
- Legacy and experimental modules remain labeled until their public API,
  schemas, and configuration contracts are stable.

## Operator Checklist

- Read `docs/README.md`, `docs/MODULES.md`, and `docs/STABILITY.md`.
- Start from a clean 2.0 tree.
- Recreate local `flight_sheet.php` and config from examples.
- Move app-specific behavior out of Dataphyre internals where possible.
- Replace dependencies on removed public modules with app-owned integrations.
- Update docs and scripts that referenced root-level v1 docs.
- Run focused app verification.
- For public release work, generate and verify `RELEASE_MANIFEST.json`.
